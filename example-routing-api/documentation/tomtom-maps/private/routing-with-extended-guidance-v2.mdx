## Purpose

The Routing with Extended Guidance v2 API provides route planning and guidance along the planned route with a new set of features.
It is conceptually and semantically similar to <a href="/routing-api/documentation/Example-maps/private/routing-with-extended-guidance" target="_blank">Routing with Extended Guidance</a>,
whereas the differences are described in this document in detail.

## Deprecation Policy

<Blockquote type="announcement" hasIcon>

This is a private API. As long as it is in private preview there is no deprecation period for modifications such as adding, removing, and modifying fields and values.
Once the API is out of private preview the deprecation period will be at most 1 year.

</Blockquote>

## Request data

### HTTPS methods: `GET`&nbsp;`POST`

- Constants and parameters enclosed in curly brackets &#123; &#125; **must be replaced** with their values.
- Please see the following <a href="#extensions-of-existing-features" target="_blank">Extensions of existing features</a> section with its parameters. The generic request format is as follows.

<h4 style={{ fontSize: '1em' }}>Generic request format</h4>

The documented endpoint extends the endpoint described in <a href="/routing-api/documentation/routing/calculate-route" target="_blank">Calculate Route</a> and supports all of its parameters.

**Note:** Only new parameters and parameters that behave differently from the ones listed in <a href="/routing-api/documentation/routing/calculate-route" target="_blank">Calculate Route</a> and <a href="/routing-api/documentation/Example-maps/private/routing-with-extended-guidance" target="_blank">Routing with Extended Guidance</a> are explicitly listed.

**Note:** A previous version of this page used <code>calculateRouteWithExtendedGuidance</code> instead of <code>calculateRoute</code> as an example for the query. Since <code>calculateRouteWithExtendedGuidance</code> is deprecated, it is strongly advised to use <code>calculateRoute</code>.

```shell [type=get] [title=URL request example]
https://{baseURL}/routing/{versionNumber}/calculateRoute/{routePlanningLocations}/json?key={Your_API_Key}
&guidanceVersion=2
&instructionsType=coded
&guidanceCoverage={string}
&sectionType={sectionType}
&instructionPhonetics={string}
&instructionRoadShieldReferences={string}
&roadAndExitIconReferencesVersion={string}
&extendedRouteRepresentation={extendedRouteRepresentation}
```

      <th scope="row">
        <code>guidanceVersion</code>
        <br />
        <em>number</em>
      </th>
      <td>
        Allows selection of the API version for Guidance.
        <p>Possible values are:</p>
        <ul>
          <li>
            <code>1</code>: Provides standard guidance features.
          </li>
          <li>
            <code>2</code>: Provides improved guidance features, but there are differences in existing features with respect to 1. Please see the following notes.
          </li>
        </ul>
        <p>
          <b>Default value:</b> <code>1</code>.
        </p>
        <Blockquote>
          <b>Notes:</b>
          <br />
          <p>
            To make use of features documented in this document,
            <code>guidanceVersion</code> must be set to <code>2</code>.
          </p>
          <p>
            Value <code>2</code> of <code>guidanceVersion</code> can
            <b>only</b> be used in conjunction with
            <code>instructionsType</code> equal to <code>coded</code>. Please refer
            to <a href="/routing-api/documentation/routing/calculate-route" target="_blank">
              Calculate Route
            </a> documentation for details on the <code>instructionsType</code> parameter.
          </p>
          <p>
            With <code>guidanceVersion=2</code> there are some changes in the
            feature set provided in the Response:
          </p>
          <ul>
            <li>
              Instruction groups are <b>not</b> provided, therefore the
              <code>instructionGroup</code> field is absent from the Response.
            </li>
            <li>
              Instruction text messages are <b>not</b> provided, therefore the
              <code>message</code> and the
              <code>combinedMessage</code> fields are absent from the Response.
            </li>
            <li>
              Whether the instruction is combinable or not should be decided by
              the client based on static road attributes and dynamic navigation
              properties. Therefore the <code>possibleCombineWithNext</code>
              field are absent from the Response.
            </li>
            <li>
              Instruction announcement points are <b>not</b> provided. The
              <code>instructionAnnouncementPoints</code> parameter can only be
              set to <code>none</code>.
            </li>
            <li>
              <code>travelMode</code> must be <code>car</code>, <code>motorcycle</code>,
              or <code>truck</code>.
            </li>
          </ul>
        </Blockquote>
      </td>
    </tr>

<table>
  <thead>
    <tr>

        Specifies the extent ("coverage") of the route for which guidance instructions will be returned.
        <p>Possible values are:</p>
        <ul>
          <li>
            <code>complete</code>: Provides guidance instructions for the whole route.
          </li>
          <li>
            <code>firstIncrement</code>: Provides sufficient guidance instructions for a route prefix to begin navigating the route.
            To obtain the instructions for the whole route, the user should then, at a later point, again call the API with <code>guidanceCoverage</code> set to <code>complete</code>.
            (Note however the restriction in the notes below).
          </li>
        </ul>
        <p>
          <b>Default value:</b> <code>complete</code>.
        </p>
        <Blockquote>
          <b>Notes:</b>
          <br />
          <p>
            It is not possible to specify the parameter <code>guidanceCoverage</code> (with any value) together with <code>supportingPoints</code> (for a description
            of <code>supportingPoints</code> see the Post Data section of the <a href="/routing-api/documentation/routing/calculate-route" target="_blank">Calculate Route</a> document).
            In particular, because route updates typically use <code>supportingPoints</code>, they are often not compatible with specifying <code>guidanceCoverage</code>.
          </p>
        </Blockquote>
      </td>
    </tr>
  </tbody>
</table>

### Structure of the `instruction` object

This section describes the `instruction` object when `guidanceVersion` is `2`. Note that this definition
overrides the `instruction` definition in the <a href="/routing-api/documentation/routing/calculate-route" target="_blank">Calculate Route</a> documentation.

<Blockquote type="announcement" hasIcon>

<h4 style={{ fontSize: '1em' }}>Guidance instruction deprecation policy</h4>
<p>
  As part of the continuous improvement of the Routing API, we also extend our
  guidance instructions with new features.
</p>
<p>
  In many cases, this requires the introduction of new maneuver codes for new
  types of instructions. Hence we cannot guarantee the same deprecation policy
  for guidance instructions as compared to the rest of the Routing API.
</p>
<p>
  In particular, clients are expected to be forward compatible, and we reserve
  the right to extend all enums defined below with new values. This includes
  completely new maneuver codes for new types of instructions.
</p>
<p>
  At the same time we reserve the right to retire enum values for instructions
  that are superseded by their improved counterparts no longer generated by the
  internal Guidance component. The same rules for
  forward-/backward-compatibility apply to fields for guidance instructions.
</p>
<p>
  For the retirement of instructions that are no longer relevant, we give a
  deprecation period of 4 months. After this period we reserve the right to
  remove deprecated values at our convenience.
</p>

</Blockquote>
